Coding Standards

In one line: Coding standards = agreed rules on how code should be written so it looks the same no matter who wrote it.
What are Coding Standards?

A defined style of writing code that every developer in an organization must follow. Code that doesn't follow these standards gets rejected during code review.

Why They Matter

  • Makes all code look uniform — doesn't matter who wrote it
  • Improves readability and maintainability
  • Reduces complexity and helps detect errors early
  • Encourages code reuse
  • Makes programmers more efficient overall
Common Coding Standard Rules
  • Limited use of globals — avoid using global variables unless necessary
  • Standard headers — each module/file must have a standard header comment block
  • Naming conventions — fixed rules for naming variables, constants, functions (e.g. camelCase, UPPER_CASE for constants)
  • Indentation — consistent spacing/tabs to show code structure
  • Error handling — standard way to return error values and handle exceptions
02

Software Reviews

In one line: Reviews = humans manually checking code/documents before testing even starts, to catch errors early and cheaply.
What is a Software Review?

A systematic inspection of software by one or more people to find and fix errors early in the development process. Done manually. Can review requirements, designs, code, test plans, etc.

Objectives

  • Improve team productivity
  • Make testing cheaper and faster (fewer bugs reach testing)
  • Deliver software with fewer defects
  • Remove problems before they become expensive

1. Desk Check

Developer traces through their own code line-by-line on paper or screen — simulating how the program runs mentally. Very early, done before even compiling.

2. Walkthrough

Author reads the code aloud to the team. Others note defects and suggestions. Informal — no management involved. Started by the code's author.

3. Code Review (Technical Review)

Peers (not management) review the code for defects. Less formal than inspection. Focus is on quality assurance. Varies from informal to semi-formal.

4. Inspection

Most formal type. Led by a trained moderator. Reviewers study the document BEFORE the meeting. Formal follow-up is done. Most effective at finding defects.

Informal Reviews

  • Desk Check — solo, no meeting
  • Walkthrough — author-led, team present
  • No strict process
  • No formal authority

Formal Reviews

  • Inspection — trained moderator leads
  • Docs reviewed BEFORE meeting
  • Formal process with follow-up
  • Most thorough and effective
⚡ Exam tip: Remember the order of formality: Desk Check (least formal) → Walkthrough → Code Review → Inspection (most formal). The key difference: Inspection uses a trained moderator and pre-meeting preparation.

Testing Strategy

Key distinction: Testing = finding errors. Debugging = fixing them. They are different activities. Testing can account for up to 40% of total project cost.
Verification vs Validation

Verification

Are we building the product right?

Checks that the software correctly implements a specific function.

Validation

Are we building the right product?

Checks that the software actually meets what the customer wanted.

The 4-Level Testing Spiral (Inside → Outside)

Testing starts with small individual pieces and grows outward to the full system.

① Unit Testing — test each individual module/function
② Integration Testing — test modules combined together
③ Validation Testing — test against customer requirements
④ System Testing — test the full system including hardware
Characteristics of a Good Testing Strategy
  • Uses Formal Technical Reviews (FTR) — catch errors before testing
  • Starts at component level, moves to full system
  • Different techniques used at different stages
  • Done by both the developer AND an independent test group
  • Must include debugging as part of the process
04

Unit Testing

In one line: Test each small piece (function/module) of software individually — in isolation from the rest.
What Unit Testing Checks
  • Interface — does data flow in and out of the module correctly?
  • Local data structures — are variables stored and used correctly?
  • Boundary conditions — does it handle edge cases (min, max, zero)?
  • Independent paths — does every logical path through the code get tested?
  • Error handling paths — does it handle errors gracefully?
Key fact: Software almost always fails at its boundaries. Always do boundary testing.
Unit Test Environment — Driver & Stubs

When you test a module, it may call other modules that aren't ready yet. So you use:

  • Driver — a fake "caller" that feeds test inputs to the module being tested
  • Stub — a fake version of a module that the tested module calls (returns dummy data)
Think of it as: Driver = boss giving work. Stub = fake employee doing nothing but returning "ok".
05

Integration Testing

In one line: Individual modules work fine alone — but do they work when connected together? Integration testing finds interface errors between modules.
3 Integration Strategies
Top-Down Start from the top module (main control). Test it using stubs for lower modules. Replace stubs one by one as you go deeper. Good for testing high-level logic early.
Bottom-Up Start from the lowest modules (no dependencies). Group them into clusters. Use drivers to test them. Combine clusters upward. Good for testing utility/low-level functions early.
Sandwich Combination: Top-down for high-level modules + Bottom-up for low-level modules. Best of both worlds. Most practical for large systems.
⚡ Exam tip: Top-down uses stubs. Bottom-up uses drivers. Sandwich uses both. This is a very common exam question.

Validation & System Testing

Validation Testing

Tests whether the software meets what the customer actually asked for (SRS document). High-level tests against functional and behavioural requirements.

Alpha Testing

  • Done at the developer's site
  • End users use the software in a controlled environment
  • Developer observes and records errors

Beta Testing

  • Done at the customer's site — real-world conditions
  • Environment is NOT controlled
  • Users record problems and report to developer
  • Developer fixes and releases final product
System Testing — 4 Types
1 Recovery Testing — Forces the system to fail and checks if it recovers correctly within an acceptable time. Evaluates MTTR (Mean Time To Repair).
2 Security Testing — Tester plays the role of a hacker trying to break in. Checks that the system resists unauthorized access. Goal: make penetration cost more than the value of the data.
3 Stress Testing — Runs the system with abnormal amounts of data, requests, or load. Tests robustness — how does it behave under extreme conditions?
4 Performance Testing — Tests runtime performance within an integrated system. Measures speed, throughput, response time. Requires both hardware and software instrumentation.

Black Box Testing

In one line: You don't care about the code inside. You only check: given this input, do I get the right output?

⬛ Black Box

  • Also called behavioural testing
  • Tests what the system does (functions)
  • No knowledge of internal code needed
  • Done in later stages of testing
  • Finds: missing functions, wrong outputs, interface errors, database errors

🔲 White Box

  • Also called glass box testing
  • Tests how the system works internally
  • Needs full knowledge of code structure
  • Done in earlier stages of testing
  • Finds: logic errors, missed paths, bad data handling
Black Box Testing Methods

1. Graph-Based Testing

Create a graph of important objects and their relationships. Design test cases that "cover" every node and link in the graph. Every object and relationship must be tested at least once.

2. Equivalence Partitioning

Divide all possible inputs into groups (classes). Pick one test case from each class. Classes have valid and invalid ranges.

Example: Input range is 1–10.
Valid class: 1 ≤ n ≤ 10 → test with n=5
Invalid class 1: n < 1 → test with n=0
Invalid class 2: n > 10 → test with n=11

3. Boundary Value Analysis

Test values at the edges of equivalence classes, not just the middle. Software almost always fails at boundaries.

Example: If 0.0 ≤ x ≤ 1.0
Valid: 0.0 and 1.0
Invalid: -0.1 and 1.1

4. Orthogonal Array Testing

Used when there are multiple input parameters each with multiple values. Instead of testing ALL combinations (which is too many), it selects a smart subset that still covers all pairs.

Example: 3 inputs (A, B, C), each with 3 values = 27 possible test cases. Orthogonal array reduces this to just 9 test cases.

White Box Testing

In one line: You look inside the code. Your goal is to make sure every path through the code gets executed at least once.
Technique 1: Basis Path Testing

Proposed by Tom McCabe. Guarantees every statement in the program is executed at least once.

Steps:

1Draw a flow graph from the program's flowchart.
2Calculate Cyclomatic Complexity V(G) — this tells you how many independent paths exist.
3Identify the independent paths (as many as V(G) says).
4Create test cases that force each path to execute.

Cyclomatic Complexity Formula

V(G) = E − N + 2

Where E = number of edges, N = number of nodes in the flow graph.

Key insight: Higher V(G) = higher chance of errors. Modules with very high V(G) need extra attention during testing.
Technique 2: Control Structure Testing

a) Condition Testing

Tests every logical condition (if, while, boolean expressions) in a module for both true and false outcomes. Catches: operator errors, variable errors, logic errors.

b) Data Flow Testing

Tracks where variables are defined (assigned a value) and where they are used. Creates a definition-use graph and ensures every definition-use (DU) chain is tested at least once.

  • DEF = variable on the left side of an assignment (e.g. x = 5)
  • USE = variable being read or used (e.g. y = x + 1)

c) Loop Testing

Focuses specifically on whether loops work correctly. Four types of loops:

  • Simple loops — one loop, test boundaries
  • Nested loops — loops inside loops
  • Concatenated loops — loops one after another
  • Unstructured loops — complex/irregular loop structure

Simple Loop — Test Cases (n = max passes):

  • Skip the loop entirely (0 passes)
  • Only 1 pass through the loop
  • 2 passes through the loop
  • m passes where m < n (some middle value)
  • n-1, n, and n+1 passes (boundary values of the loop)

Nested Loop Strategy:

  • Start at the innermost loop, hold outer loops at minimum
  • Test inner loop fully, then move outward one level at a time
⚡ Exam tip: V(G) = E - N + 2 is a formula you MUST memorize. Also know that V(G) = number of independent paths = minimum number of test cases needed for basis path testing.

Debugging

Remember: Testing finds errors. Debugging finds the CAUSE of errors and fixes them. These are different activities.
Why Debugging is Hard
  • The symptom (where you see the error) and the cause (where it actually is) can be in completely different places
  • A symptom may disappear when you fix something else
  • The cause might be a combination of things that are individually correct
  • The symptom might appear only sometimes (intermittent bugs)
3 Debugging Strategies
1 Brute Force — Dump memory, add print statements everywhere, trace everything. Most common but least efficient. Last resort. Wastes time and effort.
2 Back Tracking — Start from where the error appears and trace the code backwards to find where it went wrong. Useful for small programs.
3 Cause Elimination — List all possible causes. Devise a hypothesis for the most likely cause. Test it. Eliminate causes one by one using binary partitioning (halve the search space each time).

Two Possible Outcomes of Debugging

  • Cause found and corrected ✅
  • Cause NOT found ❌ (back to square one)
After Finding the Bug — Ask These 3 Questions
Q1Is the same bug pattern repeated elsewhere in the program?
Q2What new bug might my fix accidentally introduce?
Q3How could we have prevented this bug in the first place? (Fix the process, not just the code)

Other Testing Types

Regression Testing

When you fix a bug or add a feature, you re-run old tests to make sure you didn't break something that was working before.

  • Re-executes a subset of previously passed tests
  • Ensures changes don't introduce unintended side effects
  • Can be done manually or with automation tools
When to run it: Any time code is changed — bug fix, new feature, refactoring.
Smoke Testing

A quick sanity check — does the build even work at the most basic level? Used for "daily builds" in product development.

  • Components are integrated into a "build"
  • A small set of tests check that the build doesn't crash or fail fundamentally
  • Uncovers "show stopper" errors early
  • The entire product is smoke tested daily
  • Integration approach can be top-down or bottom-up
Comparison Testing

Used when software reliability is critical (e.g., aircraft systems, medical devices).

  • Multiple separate teams independently build the same software from the same spec
  • All versions are tested with the same inputs
  • Outputs are compared — they must all match
  • All versions run in parallel and results are compared in real-time
Model-Based Testing

Uses a behavioral model (like a state diagram) to derive test cases automatically.

1Analyze an existing behavioral model or create one (e.g., state machine).
2Traverse the model — identify inputs that cause state transitions.
3Note expected outputs at each state transition.
4Execute the test cases.
5Compare actual vs expected results and fix any differences.
Object-Oriented Testing
  • Starts by checking correctness of analysis and design models
  • The "unit" is broader — it's an entire class (not just one function)
  • Integration focuses on classes executing together across a thread or use scenario
  • Validation uses conventional black box methods
